# Conceptual Review: FlightProject Architecture (March 2026)

This document synthesizes the architectural review and technical synthesis conducted on March 12, 2026. It formalizes the relationship between the project's "Documented Intent" and the hardware realities of high-performance simulation.

## 1. The Core Architectural Paradigm: The Self-Similar Tree

FlightProject is not merely an Unreal project or a compiler; it is a **schema-bound execution system**. Every system—from a single reflected field to a world-wide orchestration plan—follows a self-similar structural fractal:

1.  **Description**: Authored intent (Vex source, Actor assets, Config rows).
2.  **Contract**: A typed schema (Vex Symbols, SIMD/GPU contracts).
3.  **Binding**: Resolving the contract against a concrete context (Mass Fragments, GPU Buffers).
4.  **Lowering**: Specializing the intent for a specific execution domain (SIMD, Verse, HLSL).
5.  **Execution**: The hardware-explicit state mutation.
6.  **Report**: A structured artifact for observability and decision-making.

**Design Rule**: To maintain system integrity, new features should never bypass the **Contract** or **Report** stages. Logic that is not "Reportable" or "Contract-Bound" is considered "Too Implicit."

## 2. Hardware-Sympathetic Data Flow

A critical design consideration is the trade-off between **Arithmetic Intensity** and **Memory Pressure** in the SIMD/Mass pipeline.

### 2.1 Kernel Fusion vs. Buffer Pass-Through
When processing data across multiple buffers (e.g., Fragment A + Fragment B + Fragment C -> GPU Storage), the project prioritizes **Simultaneous Ingestion (Kernel Fusion)**.

*   **Priority**: Process "Single-Row" (vector-wide) operations across all required buffers in a single loop.
*   **Rationale**: Memory bandwidth is the primary bottleneck. By ingesting buffers simultaneously into SIMD registers (YMM/ZMM), we maximize the "Arithmetic Intensity" (ops per byte) and minimize cache line evictions.
*   **Constraint**: Monitor **Register Pressure**. If an operation chain exceeds the available physical registers (16 for AVX2), consider breaking the kernel only at natural "Frozen" boundaries.

### 2.2 The Zero-Copy GPU Pipeline
The integration of `io_uring` and Mass Fragments allows for a "Zero-Copy" path:
1.  **Melt**: Pull data from Unreal Actors into Descriptors.
2.  **Simultaneous Ingest**: SIMD Kernel reads multiple Mass Fragments.
3.  **Direct Write**: SIMD Kernel writes directly into a Mapped GPU Storage Buffer.
4.  **Signal**: Use the `io_uring` bridge to wake the GPU without expensive CPU-side polling.

## 3. Systems as "Optic-Like Projections"

Data ownership is replaced by **Structural Projection**:
*   **Lenses/Symbols**: A system does not "own" a member variable; it uses a **Symbol** (a Lens) to focus on a piece of shared state defined by the **Schema**.
*   **Traversals/Queries**: Mass ECS queries are treated as **Functional Traversals** over heterogeneous storage.
*   **Sympathetic Intertwining**: Systems (Navigation, Swarm, Physics) are "intertwined" not through direct header dependencies, but through **Sympathetic Schemas** that allow multiple backends to project different focuses onto the same memory.

## 4. Operational Boundaries

### 4.1 Behavior Composition
*   **Atomic Behaviors**: Reusable, compiled semantic units (e.g., `Avoidance`).
*   **Composite Behaviors**: Execution-owned graphs (e.g., `Sequence`, `Selector`).
*   **The Orchestration Split**: Orchestration decides *legality* (which behavior is valid for a cohort), while the Behavior itself decides *runtime branching* (e.g., "Shield low? -> Flee").

### 4.2 The "Melt/Freeze" Lifecycle
*   **Melted (Authoring)**: Mutable, reference-heavy Unreal Actors and Assets.
*   **Frozen (Simulation)**: Contiguous, SoA-aligned Mass Fragments and GPU resources.
*   **The Bridge**: Subsystems must explicitly manage the transition from Melted to Frozen state to ensure thread safety and cache coherency.

## 5. Summary Design Heuristic

When designing or reviewing a system in FlightProject, ask:
1.  **Intent**: What is the described goal?
2.  **Contract**: What Schema defines the legal symbols and types?
3.  **Binding**: Where is the data "Frozen" (Mass, GPU, or CPU Heap)?
4.  **Lowering**: Which hardware backend (SIMD, Verse VM, or HLSL) is most sympathetic to this data?
5.  **Report**: How does the system explain its decisions back to the Orchestrator?
